Deal Page Enhancement

To help users to quickly evaluate a deal

Project Brief

Many of Groupon users were struggling with quickly finding the information they needed to evaluate a deal. Since the page is “deal-centric”, the content structure didn’t match user's shopping mental model. It emphasized low discounts and prices, yet merchant information that would highly impact the purchase decision for local deals. Additionally, users sometimes had a hard time understanding the deal itself at a glance since the option title was extremely lengthy. In this project, I focused on these two main problems users are facing during the quick heuristic evaluation stage.

Skills
  • UX Design
  • Visual Design
  • Prototyping

My Role

I was the sole designer and worked with a PM. At the initial phase, I co-hosted a brainstorming section with the leadership and biz ops people to have them align with problems we want to solve. Afterward, we broke down the problem into two focuses. In each focus, I used past user research takeaways and existed data to drive the design decision.

Scope and Problems

From past user studies, we found that users need to put a lot of effort to evaluate a deal, although they were just browsing. That’s because the content structure was “deal-centric” and that didn’t match users shopping mental model. Many important information is hard to find at a glance. In this project, I would focus on addressing the two main problems in the quick evaluation stage.

Two biggest problems in quick heuristic evaluation stage
  1. Critical merchant infomation is hard to access. For example, merchant name and location are either too small to see or hidden at the bottom of the page.
  2. Option title is too lengthy to parse. When there are multiple deal options, the option titles are hard to compare the differences.
What’s the success will look like
  • Users are able to quickly decide if this is the deal for them and have the information they need at their fingertips.
  • User can easily compare different options and find the best option for them.

Brainstorm to get people on the same page

1

Comprehend a Deal at a Glance

When evaluating a local deal, users are not only looking for the deal title and the price but also key merchant info and how good this deal is. The current deal-driven page didn’t help users to find them easily.


Problems
  1. Repeated contents in deal title and option title. Repetition doesn’t add more meaning. Instead it clutters the page and slow down users in processing information.

  2. Critical merchant info, such as location, is hard to access. Users have to scroll to the bottom of the page to find the map.

  3. Repetitive and robotic sounding urgency messages clutter the deal page. It’s lack of emotional connection, and message like “limited time remaining! ” is vague and meaningless.

Design Proposal

(1) Merchant-centric Page Header

Users want to know: What's about the merchant?

One of the highest clicked links on the deal page was the merchant website (although it was at the bottom of the page). That behavior showed a strong intent of accessing of merchant information. Also, location and review impact users purchase decision the most at an early stage based on the research. Therefore, I created this merchant-centric page header to help the user to find merchant info easily.

(2) Dynamic Conversational Urgency Message

Users want to know: How good is this deal?

When looking at a deal provided by an unknown merchant, urgency message will give users an idea of how good the deal is. However, the current message was easily to be ignored. In order to improve the awareness of the urgency without making it irritating, I limited it to show one message at a time and utilized a conversational tone to make it more meaningful for the user.

2

Improving the scalability of the option

Complex option title will take users longer time to comprehend, especially when the option title contains too many details, such as time, location, what’s included...etc. In other cases, some deals options are too similar, that makes users harder to find the right option at a glance.


Problems
  1. Lengthy and trivial option title. Users need to spend longer time to read the option title and it’s difficult to compare between options.

  2. Similar options title can easily cause purchase mistakes because there’s only a few words difference between options

Design Direction

To make the option easy to digest and avoid wrong purchases, the idea is to break down option title to key attributes. It will create speed bumps to force user pay attention on key decisions. Meanwhile, it transforms some restrictions, such as dinner only, to an option, which can make the option feels less restrictive.

First Iteration (2018 March)

Failed 😢

In this version, we were using the existing Groupon half sheet design pattern, however, the experiment failed. So I went back to dig into the user behaviors and data to found a couple issues in this version.

Issues:

  • The deal options are hidden behind traits. Data shows that displaying option content ahead has positive impact to conversation.
  • It’s hard to compare the price between options since the price is too small to be noticed.
  • The “box” UI element limits the text length of the option. It’s not scalable for long option title.
  • The primary “Buy” button makes user trigger error case easily.
  • When the option text is longer than two lines, the rest of the text will be cut off, so users can’t read the full option title.

Design Explorations

#1: Exposed Options

Implementation difficulty: ★☆☆☆☆

Pros:
  • Users are able to see all options at one place.
  • Users can receive immediate feedback when their selection impacts the availability of options in another trait. In this example, All Ages(3+) is not available for 1-Day General Admission option, so All Ages(3+) will be greyed out once 1-Day General Admission is selected.

Cons:
  • The “pill” UI doesn’t scale well to long text option title.
  • More options could be hidden since user need to scroll horizontally to discover them.
  • The default price is the lowest price but user’s don’t know it associates with what combination of options.
  • It’s hard to compare the price between options/combinations since users can only see one price at the time.

#2: Exposed Price with Options

Implementation difficulty: ★★☆☆☆

Pros:
  • The list format is able to scale to long text option title.
  • Users are able to see the price with each option side by side in the half-sheet.
  • Use “Select” to replace “Buy” when selection hasn’t completed, so users won’t see the error message.

Cons:
  • Users are not able to see the options directly on the deal page.
  • Can’t see how the selection impacts another trait since users can only see one trait at a time.
  • Hard to compare the final price in the middle of the selection.

#3: Create hierarchy between traits and expose price with options

Implementation difficulty: ★★★★★

Pros:
  • User will only see the key offer be exposed on the deal page. For doing that, we need to define the key offer and attributes. In this case, Admission Type is the key offer, and Travel Period and Guest Age are attributes under it.
  • Users are able to easily access additional info. In this case, there’s a link next to each Travel Period to explain when is it.
  • The final selection and price is called out with buy button so users can confirm the order easily

    • Cons:
    • Require back-end tool change and editorial team’s training to support this setup
    • Remove Buy button from the deal page could potentially have negative impact to conversation

    Implementation

    Considering the technical restriction, we picked #1 and #2 to run an A/B testing on mobile web and Android Local deal page.

    Takeaways from the testing result:
    • #2 performs better than #1
    • The new design helps users to avoid purchase mistakes on Local deal page. The refund rate reduce 0.18% in 30 days
    • User make less purchase when display option in traits on Local deal page. It’s opposite to Goods and Travel deal page result.
    Next steps for the iteration:

    Traits doesn’t work on all Local deals since it causes user miss potential options. But for some complex ticket deals with a massive number of options, it does help the user to find the potential option faster. Therefore, we should work closely with customer support and data scientists to define what are the right Local deal types to apply traits.
